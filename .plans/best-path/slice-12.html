Slice 12 — SSE state stream

P0Delivery #12Local integration

Broadcast full state only after commits; full snapshots are simpler than deltas.

Best path forward

  1. Add one small RaceRuntime::apply application function around EventStore and a tokio broadcast channel; HTTP handlers and the due-event task use it, while EventStore remains transport-free.
  2. RaceRuntime publishes the full projected state only after a successful commit. The SSE handler subscribes first, then loads/emits current state, then streams newer states so no commit can fall between snapshot and subscription.
  3. Include the durable race generation and last committed protocol sequence in each full state. On connect/reconnect or lag recovery, accept higher generations and newer same-generation sequences; reject delayed older-race states. Use full snapshots, not a resumable event log or deltas.

Verification

  • cargo test sse_emits_state_after_command
  • A commit racing with SSE connection setup is observed either in the initial state or the subscribed stream
  • A lagged or reconnecting client that misses a race boundary accepts the higher generation and rejects delayed old-race states

Keep it simple: not in this slice

  • WebSockets
  • Event offsets
  • Per-client filters or deltas
