Implementation Plan — LapX

LapX is a local Carrera timing and race-control system for 1–4 lanes. The old plan put most RaceEngine, storage, CLI, HTTP, and UI work into one huge slice. This reevaluation keeps completed core work visible and breaks the remaining work into narrow, independently verifiable tracer bullets.

Reevaluation

Cut made: the original Slice 1 was too large. HTTP, SSE, debug UI, GPIO, production UI, and Verwaltung are now separate slices. Each todo slice has one demo path and one smallest useful test. Before HTTP exists, lapxctl is the checkable control surface.

Design Decisions

Keep RaceEngine UI- and transport-independent

Race rules stay in Rust domain code; CLI, HTTP, SSE, debug UI, GPIO, and SvelteKit only feed commands or read projected state.

Why: one rule path is easier to test and replay · Revisit if: multiple race formats require separate rule engines.

SQLite remains the only Rennprotokoll store

Every race event counts only after a SQLite transaction commits to the append-only race_events table.

Why: simplest durable recovery on Raspberry Pi · Revisit if: write latency is measured as race-affecting.

Use HTTP POST for commands and SSE for live display

Commands stay request/response; live state updates stream as server-sent events.

Why: less state than WebSockets and matches ADR 0002 · Revisit if: bidirectional realtime controls become necessary.

Debug page before SvelteKit production UI

The next UI should be a tiny static debug page served by Rust, not a full frontend scaffold.

Why: proves HTTP/SSE endpoints with minimal code · Revisit if: debug and production UI start duplicating too much code.

Hardware behind small seams

GPIO input and relay output adapters sit outside RaceEngine and use the same command/state paths as simulation.

Why: simulation remains testable off-Pi · Revisit if: interrupt timing forces a lower-level capture queue.

Public API is read-only

Mutating endpoints bind locally; public WLAN clients get display state only.

Why: avoids accidental remote race control · Revisit if: authenticated remote control is explicitly required.

Ticket-ready design notes

Build LapX as thin tracer bullets. The core RaceEngine/Rennprotokoll/CLI path is already split from transport. Next ticket should start at Slice 10: add a local Rust HTTP state endpoint backed by the existing SQLite EventStore and RaceEngine projection. Do not add SvelteKit, GPIO, auth, or a command bus in that ticket; those are later slices with their own verification.
